# 🦀 Crab

**Crab** — простой и быстрый сборщик для проектов на **C и C++**, написанный на Rust.
По духу похож на `cargo`: создаёт проекты, инкрементально собирает их в debug/release,
запускает бинарник и модули, собирает статические/динамические библиотеки и умеет
показывать дерево зависимостей `#include`.

---

## Возможности

- 📦 **Создание проектов** одной командой (`crab new`) и инициализация существующих (`crab init`).
- ⚡ **Инкрементальная сборка** — перекомпилируются только изменённые файлы. Изменения
  отслеживаются и в исходниках, и в **заголовках** (через `gcc -MM`).
- 🧵 **Параллельная компиляция** (rayon) — файлы собираются одновременно.
- 🧩 **Модули** — отдельные подкаталоги собираются в самостоятельные бинарники.
- 📚 **Библиотеки** — сборка статических (`.a`) и динамических (`.so`).
- 🔎 **Поиск сторонних библиотек** — автоматически по системным путям или вручную.
- 🌳 **Дерево зависимостей** `#include` (`crab tree`).
- 🧪 **Тесты** — сборка и запуск тестовых бинарников из `tests/` (`crab test`).
- 🎨 **Форматирование кода** через `clang-format` (`crab fmt`).
- 🧠 **Интеграция с IDE** — генерация `compile_commands.json` для clangd/VS Code/CLion (`crab compdb`).
- ⚙️ **Пользовательские флаги сборки** через секцию `[build]` в `config.toml`.
- 🧹 **Автоочистка** устаревших объектных файлов от удалённых исходников.

---

## Требования

- **Компилятор C/C++**: `gcc` (для C), `g++` или `clang` (для C++).
- **`ar`** — для сборки статических библиотек.
- **`clang-format`** — опционально, для команды `crab fmt`.
- Работает на **Linux, macOS и Windows** с тулчейном семейства gcc (на Windows —
  например, MinGW). Расширения подбираются по платформе: исполняемые
  (`.exe` на Windows) и динамические библиотеки (`.so` / `.dylib` / `.dll`).
- Автопоиск сторонних библиотек работает на всех платформах: каталоги
  заголовков и библиотек запрашиваются у самого компилятора
  (`gcc -print-search-dirs` и т.п.), поэтому находит и в MSYS2/MinGW на
  Windows. Нестандартные расположения можно добавить через
  `[libraries].path` в `config.toml`.

Для сборки самого Crab нужен **Rust** (edition 2024).

## Установка

Соберите из исходников:

```bash
git clone https://github.com/atxxxm/Crab
cd Crab
cargo build --release
```

Готовый бинарник будет в `target/release/crab` — положите его в каталог из `PATH`.
Либо установите напрямую:

```bash
cargo install --path .
```

## Быстрый старт

```bash
crab new myapp --lang cpp      # создать проект на C++
cd myapp
crab build                     # собрать (debug)
crab run                       # собрать и запустить
crab build release             # оптимизированная сборка
```

Запуск без аргументов (`crab`) показывает справку. Подробная справка по любой
команде — через `--help`, например `crab build --help`.

## Команды

### `crab new <NAME>` — создать новый проект

| Опция | Описание |
|-------|----------|
| `-l, --lang <c\|cpp>` | язык проекта (по умолчанию `cpp`) |
| `-g, --git` | инициализировать git-репозиторий и добавить `.gitignore` |
| `-c, --cli` | шаблон `main` с аргументами (`int argc, char *argv[]`) |
| `--lib` | шаблон библиотеки вместо бинарника (без `main`, создаёт `src/<name>.cpp` + `include/<name>.hpp`) |

```bash
crab new myapp
crab new mylib --lang c --git
crab new tool --cli
crab new mylib --lib              # библиотека на C++
crab new mylib --lib --lang c     # библиотека на C
```

После `crab new mylib --lib`:

```text
mylib/
├── config.toml
├── crb/
├── src/
│   └── mylib.cpp    # реализация
└── include/
    └── mylib.hpp    # публичный заголовок
```

Собрать как библиотеку:

```bash
cd mylib
crab build lib static    # -> crb/library/static/libmylib.a
crab build lib dynamic   # -> crb/library/dynamic/libmylib.so
```

### `crab init` — инициализировать проект в текущей папке

Собирает существующие `.c/.cpp/.h/.hpp`, раскладывает их по `src/` и `include/`,
создаёт `config.toml` и каталог сборки.

### `crab build` (алиас `b`) — собрать проект

| Подкоманда | Описание |
|------------|----------|
| *(без аргументов)* / `debug` | сборка в debug (по умолчанию) |
| `release` | оптимизированная сборка |
| `module <NAME> [-r]` | собрать модуль (`-r` — в release) |
| `lib <static\|dynamic>` | собрать библиотеку |

```bash
crab build
crab build release
crab build module net -r
crab build lib static
```

Флаги по умолчанию: **debug** — `-g -O0 -Wall -Wextra -pedantic`,
**release** — `-O2 -flto` (и `-s` при линковке).

### `crab run` — собрать и запустить

Перед запуском проект **инкрементально пересобирается** (как `cargo run`).

| Опция | Описание |
|-------|----------|
| `-r, --release` | запустить release-сборку |
| `-m, --module <NAME>` | запустить модуль вместо основного бинарника |
| `--gdb`, `--valgrind` | пробросить соответствующий флаг в программу |
| `-- <ARGS>` | аргументы, передаваемые программе |

```bash
crab run
crab run -r
crab run -m net
crab run -- --port 8080
```

### `crab clean` (алиас `c`) — удалить артефакты сборки

| Подкоманда | Описание |
|------------|----------|
| *(без аргументов)* / `all` | очистить весь каталог сборки |
| `debug` / `release` | очистить только debug/release |
| `module <NAME>` | очистить сборку модуля |
| `lib` | очистить собранные библиотеки |

### `crab config` (алиас `conf`) — настройки проекта

```bash
crab config set --lang c
crab config set --compiler clang     # gcc | gpp | clang
```

### `crab module` (алиас `m`) — работа с модулями

```bash
crab module add net       # зарегистрировать подкаталог src/net как модуль
crab module remove net
```

### `crab compdb` (алиас `cc`) — база компиляции для IDE

Генерирует `compile_commands.json` в корне проекта — его понимают clangd,
VS Code (C/C++, clangd), CLion и др., что даёт автодополнение, переход к
определению и диагностику.

| Опция | Описание |
|-------|----------|
| `-r, --release` | использовать флаги release вместо debug |

```bash
crab compdb            # сгенерировать (debug-флаги)
crab compdb --release
```

В базу попадают компилятор, `-I` (заголовки проекта и сторонние), флаги
профиля и пользовательские флаги из `[build]`. Файл стоит добавить в
`.gitignore`.

### `crab test` (алиас `t`) — сборка и запуск тестов

Ищет исходники в каталоге `tests/` (или в другом, настроенном через `[test].dir`).
Каждый файл компилируется в отдельный бинарник и запускается; выход 0 = тест прошёл.
Файл `src/main.c(pp)` автоматически исключается из линковки, чтобы тест мог определить
свой `main()`.

| Опция | Описание |
|-------|----------|
| `[FILTER]` | запустить только тесты, в имени файла которых содержится `FILTER` |
| `-r, --release` | собрать и запустить тесты с оптимизациями release |

```bash
crab test            # все тесты
crab test math       # только тесты, в имени которых есть "math"
crab test -r         # тесты против release-сборки проекта
```

Пример структуры тестов:

```
tests/
├── test_math.cpp    # содержит свой main(), вызывает assert() / возвращает 0 или 1
└── test_net.cpp
```

Вывод:
```
  Finished debug target in 0.01s
   Running 2 test file(s)
  test tests/test_math.cpp ... ok
  test tests/test_net.cpp  ... FAILED

test result: FAILED. 1 passed; 1 failed; finished in 0.03s
```

### `crab fmt` (алиас `f`) — форматирование кода

Запускает `clang-format` по исходникам и заголовкам проекта.

| Опция | Описание |
|-------|----------|
| `--check` | только проверка, без изменений (ненулевой код выхода, если есть расхождения — удобно для CI) |
| `--style <STYLE>` | стиль clang-format (`LLVM`, `Google`, `Mozilla`, …) |

```bash
crab fmt                  # отформатировать на месте
crab fmt --check          # проверить (ничего не меняя)
crab fmt --style Google
```

По умолчанию `clang-format` уважает файл `.clang-format` в проекте, а при его
отсутствии использует стиль LLVM; `--style` переопределяет это.

### Глобальные опции

- `-v, --verbose` — писать подробный лог в `crb/crab.log` (также включается
  переменной окружения `CRAB_LOG`). По умолчанию логирование выключено.
- `--no-color` — отключить цветной вывод. Цвет также отключается переменной
  `NO_COLOR` и при перенаправлении вывода (не в терминал); `CLICOLOR_FORCE`
  включает цвет принудительно.
- `-V, --version`, `-h, --help`.

## Структура проекта

После `crab new myapp`:

```text
myapp/
├── config.toml        # конфигурация проекта
├── crb/               # каталог сборки (артефакты, в .gitignore)
└── src/
    └── main.cpp       # точка входа
```

Заголовки по умолчанию ожидаются в `include/` (настраивается в `config.toml`).

---

## Конфигурация (`config.toml`)

```toml
[project]
name = "myapp"
version = "0.0.1"
created = 2026

[settings]
lang = "c++"            # "c" или "c++"
compiler = "g++"        # gcc / g++ / clang
source_dir = "src"      # каталог исходников
header_dir = "include"  # каталог заголовков

[build]                 # пользовательские флаги (всё опционально)
standard = "c++17"                       # -> -std=c++17
defines = ["DEBUG", "VER=2"]             # -> -DDEBUG -DVER=2
include_dirs = ["third_party/include"]   # -> -I...
cflags = ["-Wpedantic", "-march=native"] # доп. флаги компиляции
ldflags = ["-lpthread", "-lm"]           # доп. флаги линковки

[test]                  # опционально; по умолчанию dir = "tests"
dir = "tests"           # каталог с тестовыми файлами

[files]                 # список исходников; "on" — собирать, иное — игнорировать
"src/main.cpp" = "on"

[libraries]             # ручные пути поиска сторонних библиотек (см. ниже)
path = []

[module]                # зарегистрированные модули (заполняется автоматически)
```

Флаги из `[build]` **добавляются поверх** встроенных флагов профиля и идут
последними — значит, могут их переопределять. Секция опциональна: старые
конфиги без неё продолжают работать.

### Исключение файлов из сборки

В секции `[files]` каждому исходнику соответствует значение `"on"` (собирается).
Чтобы временно исключить файл, поставьте любое другое значение, например:

```toml
[files]
"src/main.cpp" = "on"
"src/experimental.cpp" = "off"
```

## Структура каталога сборки (`crb/`)

```text
crb/
├── data/                 # данные о найденных сторонних библиотеках
├── debug/
│   ├── obj/              # объектные файлы (.o)
│   └── bin/<project>     # исполняемый файл
├── release/
│   ├── obj/
│   └── bin/<project>
├── library/
│   ├── static/  → lib<name>.a
│   └── dynamic/ → lib<name>.so
├── module/<name>/...     # сборки модулей (та же структура debug/release)
└── crab.log              # лог (только при -v / CRAB_LOG)
```

## Как работает инкрементальная сборка

1. Crab собирает список исходников и через `gcc -MM` строит файл зависимостей
   (исходник → его заголовки).
2. Для каждого исходника сравнивается время модификации его самого **и всех его
   заголовков** с сохранённым состоянием.
3. Перекомпилируются только изменившиеся файлы; правка заголовка вызывает
   пересборку всех зависящих от него исходников.
4. Перед линковкой удаляются объектные файлы, для которых больше нет исходника.

## Модули

Модуль — это подкаталог в `src/`, собираемый в отдельный бинарник.

```bash
crab module add net          # src/net/ становится модулем
crab build module net        # собрать (crb/module/net/debug/bin/net)
crab run -m net              # собрать и запустить модуль
crab clean module net        # очистить сборку модуля
crab module remove net
```

## Сторонние библиотеки

Crab анализирует `#include <...>` в коде, отбрасывает стандартные заголовки и
ищет оставшиеся библиотеки:

- **Автоматически** — по каталогам поиска самого компилятора
  (`gcc -print-search-dirs`, `gcc -E -Wp,-v`), а также `./include`, `./lib`
  и переменной `CPATH`. Работает на Linux, macOS и Windows (MSYS2/MinGW).
- **Вручную** — укажите каталоги для поиска в `config.toml`:

  ```toml
  [libraries]
  path = ["/opt/SDL2", "/home/user/libs"]
  ```

Найденные пути подключаются к компиляции (`-I`) и линковке (`-L`/`-l`)
автоматически.
